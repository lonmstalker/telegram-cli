## Crash and resource ownership model

- One global `.build.lock` serializes build/reuse/recovery. Its open file description is inherited by every configure/build/preflight/inspection watchdog, but not by the guarded target.
- Watchdog writes state `starting`, starts a gated target, durably writes `running`, sends the complete `started` frame to the parent and only then releases the target gate. The parent drains fragmented `SOCK_STREAM` frames through EOF before declaring handshake failure.
- A private `.work-<id>` has an exact mode-`0600` owner marker. Startup recovery runs only after acquiring the global lock, recursively validates any nested guard state and never follows symlinks.
- Destructive cleanup atomically renames work to `.reap-<id>`. Final marker removal uses a sibling `.reap-proof-<id>.json`: proof rename, directory `fsync`, empty-directory removal, parent `fsync`, proof removal. Recovery accepts only a valid proof plus an empty reap directory or a valid orphan proof.
- Malformed work/reap/proof paths, symlinks, live watchdog/PGID state and same-ID work/reap ambiguity fail closed.

- Build-lock wait: 30 s. Stale cleanup: 60 s, максимум 2 work identities и 500 000 scanned/removed entries. Новый scratch не создаётся, пока recovery не завершён.
